# 작업 안내

주 사용자: 같이

## 이 문서는 무엇인가요?

이 문서는 지금 해야 할 일과 작업 순서를 적는 곳입니다.

쉽게 말하면:

- 지금 무엇을 먼저 해야 하는지
- 다음에는 무엇을 해야 하는지
- 작업할 때 어떤 순서로 하면 좋은지

일반적인 개발 말로 하면 이 문서는 `구현`과 `테스트`를 돕는 문서입니다.

사용자와 AI가 같이 작업 순서를 맞출 때 보는 문서입니다.

## 구현할 때 보통 하는 순서

1. 요구사항을 확인합니다.
2. 관련 설계 문서를 봅니다.
3. 코드를 고칩니다.
4. 테스트하거나 직접 실행해 봅니다.
5. RTM 상태를 업데이트합니다.
6. 문서를 같이 고칩니다.

## 지금 먼저 보면 좋은 것

- [ ] 게임이 실제로 잘 실행되는지 확인
- [ ] 소환과 전투가 잘 되는지 확인
- [ ] 화면 표시가 게임 설명과 맞는지 확인
- [ ] 테스트 실행 방법 확인

## 다음에 하면 좋은 것

- [ ] 게임 설명 문서 더 다듬기
- [ ] 설계 문서와 실제 코드 비교하기
- [ ] 배포 문서 최신 상태 확인하기

## 작업은 이렇게 하면 쉽습니다

### 1. 무엇을 바꿀지 정합니다

예:

- 게임 규칙
- 화면
- 코드 구조
- 배포 방법

### 2. 맞는 문서를 먼저 봅니다

- 요구사항: `docs/requirements/requirements.md`
- 요구사항 추적: `docs/requirements/rtm.md`
- 게임 설명, 화면 이해: `docs/game/game.md`
- 코드 구조: `docs/design/design.md`
- 배포: `docs/deploy/deploy.md`

### 3. 코드를 고칩니다

- 필요한 파일을 수정합니다.

### 4. 테스트하거나 직접 확인합니다

예:

- 게임이 켜지는지 확인
- 화면이 잘 보이는지 확인
- 전투가 잘 되는지 확인
- 테스트가 있으면 실행

### 5. RTM도 같이 확인합니다

- 구현이 시작되면 RTM 상태를 바꿉니다.
- 구현이 끝나면 RTM 상태를 바꿉니다.
- 검증까지 끝나면 `Verified`로 바꿉니다.

### 6. 문서도 같이 고칩니다

- 요구사항이 바뀌면 `docs/requirements/requirements.md`
- 설명이 바뀌면 `docs/game/game.md`
- 구조가 바뀌면 `docs/design/design.md`
- 할 일이 바뀌면 `docs/work/work.md`

## 작업을 시작하기 전에 기억할 것

- 너무 많은 것을 한 번에 바꾸지 않습니다.
- 무엇을 바꿨는지 기록하면 좋습니다.
- 헷갈리면 이 문서부터 다시 봅니다.

## 한 줄 정리

작업을 시작하기 전에는 항상 `work/work.md`를 먼저 보고,
끝날 때는 테스트와 문서 수정도 함께 합니다.
